# Nop.Plugin.Payments.Latipay

Hosted redirect payment plugin for Latipay on nopCommerce.

## Supported Version

- nopCommerce `4.90.x`
- .NET `9`
- Plugin version in [`plugin.json`](/Users/ziqiwang/MyProjects/GitRepo/znopcommerce/src/Plugins/Nop.Plugin.Payments.Latipay/plugin.json) is currently `4.90.3`

## Purpose

This plugin integrates Latipay as a hosted redirect payment method for nopCommerce. Customers choose a configured Latipay sub-payment method during checkout, are redirected to Latipay to complete payment, and the plugin confirms payment only after a verified callback and/or verified server-side reconciliation query.

The implementation is intentionally defensive:

- browser `return_url` is informational only
- payment status changes require verified signature checks and safe state transitions
- retries always reuse the same nopCommerce order
- refund history is stored in plugin-owned tables
- uncertain states remain pending or review-required instead of overstating success

## Supported Capabilities

- Hosted redirect flow only
- NZD only
- Customer sub-payment method selection during checkout
- Customer retry from order details using the same nopCommerce order
- Full refund
- Partial refund when enabled in plugin settings and accepted by the Latipay account/API
- Manual and optional scheduled reconciliation
- Redacted logging for provider interactions

## Unsupported Capabilities

- Recurring billing
- Stored cards or tokenization
- Merchant-hosted card entry
- Capture/void flows outside the documented hosted redirect pattern
- Multi-currency processing
- Multi-store configuration overrides

## Setup

1. Build and deploy the plugin into the nopCommerce plugins folder.
2. Install or reload the plugin from the nopCommerce admin area.
3. Open the Latipay configuration page in admin.
4. Enter the required Latipay credentials and API base URL for each enabled Latipay integration mode.
5. Enable at least one Latipay sub-payment method.
6. Ensure the public `return_url` and `callback_url` generated by the store are reachable by Latipay over HTTPS.
7. Test the full payment flow in a non-production environment before enabling the plugin for live customers.

## Configuration Fields

- `Enabled`: controls whether the payment method is offered during checkout
- `Use sandbox`: environment intent flag for operations and troubleshooting
- `API base URL`: configurable provider base URL for the hosted-online Latipay APIs, defaulting to `https://api.latipay.net/`
- `User ID`: required for hosted-online Latipay API requests
- `Wallet ID`: required for hosted-online transaction creation
- `API key / secret`: shared signing key used for hosted-online request signing and response verification
- `Card API base URL`: configurable provider base URL for the hosted card APIs
- `Card merchant ID`: required for the hosted card API contract
- `Card site ID`: required for the hosted card API contract
- `Card private key`: shared signing key used for hosted card request signing and callback/query verification
- `Debug logging`: enables additional redacted provider diagnostics
- `Enable refunds`: allows nopCommerce refund operations through Latipay
- `Enable partial refunds`: allows partial refunds when the account and API support them
- `Enable reconciliation task`: enables the scheduled task that rechecks unresolved payments
- `Reconciliation period minutes`: schedule interval for unresolved payment checks
- `Retry guard minutes`: minimum cooling-off period before automatically allowing another retry on uncertain states

Each sub-payment method can also be enabled or disabled individually, and each has an optional admin-facing display-name override.

## Supported Payment Methods

The plugin keeps provider-facing method values centralized and uses safe internal keys in the UI:

- `Alipay` -> `alipay`
- `WeChat Pay` -> `wechat`
- `NZ bank transfer` -> `polipay`
- `PayID` -> `payid`
- `UnionPay UPOP` -> `upi_upop`
- `Card (Visa/Mastercard)` -> `vm`

The implementation also normalizes the Latipay return/query alias `onlineBank` back to the NZ bank transfer method for verification purposes.
Hosted card support is implemented as one provider option because the official card documentation exposes the hosted card method as `vm`; the plugin does not split that into separate Visa and Mastercard choices.

## Checkout, Callback, And Return Behavior

Checkout flow:

1. Customer selects Latipay and chooses one enabled sub-payment method.
2. nopCommerce creates the order in a pending payment state.
3. The plugin creates a Latipay payment attempt and redirects the customer to the hosted Latipay URL.

Hosted card flow notes:

- The plugin submits billing/contact details from the nopCommerce billing address to the hosted card API.
- If required hosted-card payer details are missing, the payment attempt is left pending and the redirect is not started.
- Browser return for hosted card is still informational only; payment confirmation comes from callback and/or server-side query.

Callback flow:

1. Latipay posts the payment result to the plugin callback endpoint.
2. The plugin validates required fields and verifies the status signature.
3. The callback is processed under a per-attempt lock and duplicate callbacks are ignored safely.
4. A verified state machine applies legal transitions only.

Return flow:

1. The customer returns from Latipay through `return_url`.
2. The browser return is never treated as proof of payment.
3. The plugin immediately performs a server-side reconciliation query by `merchant_reference`.
4. The order is marked paid only if reconciliation or callback verification confirms the payment safely.

## Retry Behavior

- Retry is available from the order details page only for unpaid Latipay orders in a safely retryable state.
- Retry always reuses the existing nopCommerce order.
- Retry always creates a new `LatipayPaymentAttempt` row with a new `merchant_reference`.
- The customer can reselect any currently enabled Latipay sub-payment method.
- The plugin reconciles uncertain states before allowing retry where appropriate.
- Rapid duplicate retry submissions are blocked with order-level locking.

## Refund Behavior

- Refunds are routed through nopCommerce refund operations and delegated to the plugin refund service.
- Full refunds are supported when the order is paid and the confirmed Latipay transaction identifier exists.
- Partial refunds are supported when enabled in settings and still within the remaining refundable amount.
- The plugin prevents duplicate and over-limit refunds by combining:
  - order-level locking
  - plugin refund records
  - completed refund totals
  - locally reserved review-required refunds
- If a refund result is ambiguous, the plugin stores the refund as `ReviewRequired` and does not overstate success.

## Plugin-Owned Database Design

The plugin stores durable provider state only in plugin-owned tables.

### `LatipayPaymentAttempt`

Tracks each hosted payment attempt for a nopCommerce order, including:

- `OrderId`
- `AttemptNumber`
- `MerchantReference`
- selected sub-payment method
- external Latipay identifiers and status
- amount and currency
- callback verification markers
- retry linkage
- timestamps for redirect, callback, payment completion, and reconciliation
- short redacted failure summary

Important constraints and indexes:

- unique `MerchantReference`
- unique `(OrderId, AttemptNumber)`
- callback idempotency lookup index
- retry parent lookup index
- query lookup indexes by order and Latipay order ID

### `LatipayRefundRecord`

Tracks each refund attempt initiated by the plugin, including:

- `OrderId`
- `PaymentAttemptId`
- `LatipayOrderId`
- `RefundReference`
- `RefundAmount`
- `RefundStatus`
- requested/completed timestamps
- short redacted provider response summary

Important constraints and indexes:

- unique `RefundReference`
- refund history lookup by order
- lookup by payment attempt and Latipay order ID

The plugin does not store raw secrets in these tables and avoids storing raw provider payloads.

## Reconciliation Behavior

Reconciliation queries Latipay by `merchant_reference` and is used in three places:

- browser return handling
- retry eligibility checks
- optional scheduled task execution

During reconciliation the plugin validates:

- `merchant_reference`
- amount
- currency
- payment method mapping
- legal state transition rules

If the result cannot be trusted or conflicts with the stored state, the order stays pending and the attempt is marked for review rather than being forced into a paid state.

## Troubleshooting

- If Latipay does not appear at checkout, verify the plugin is enabled, at least one sub-payment method is enabled, and the customer currency is `NZD`.
- If the payment returns but the order remains pending, check whether the callback reached the store and whether reconciliation could verify the attempt.
- If retry is blocked, inspect the latest payment attempt and reconciliation notes for an unresolved or recently-created attempt.
- If refunds are blocked, confirm the order is paid, the Latipay order identifier exists, and there is enough refundable balance remaining after completed or review-required refunds.
- If provider requests fail, verify `User ID`, `Wallet ID`, `API key`, and `API base URL`.
- If hosted card requests fail, verify `Card merchant ID`, `Card site ID`, `Card private key`, and `Card API base URL`.
- If debug logging is enabled, review logs for redacted request/response diagnostics rather than sensitive credential data.

## Security Considerations

- Orders are never marked paid from browser return data alone.
- Status signatures are verified server-side using the configured shared secret for each Latipay API family.
- Payment and refund operations use locks to reduce duplicate processing.
- Duplicate callbacks are handled idempotently.
- Unknown, mismatched, or conflicting statuses leave the order pending or review-required.
- Sensitive values are redacted in provider logs.
- Secrets are stored in plugin settings and are never rendered back into the admin UI once saved.

## Assumptions From Latipay Documentation

- Hosted redirect integration follows the Latipay hosted-online documentation for `user_id`, `wallet_id`, `payment_method`, `amount`, `return_url`, `callback_url`, `merchant_reference`, `ip`, `version`, and `product_name`.
- Hosted card integration follows the Latipay hosted payment pages documentation for `merchant_id`, `site_id`, `order_id`, payer billing/contact fields, `timestamp`, `return_url`, `callback_url`, and refund/query contracts under the card-payments API.
- The plugin assumes a unique `merchant_reference` per payment attempt, even when retrying the same nopCommerce order.
- Latipay documentation uses both `polipay` and `onlineBank` for the NZ bank flow; the plugin normalizes those to the same internal method.
- Hosted card support is modeled as a single `Card (Visa/Mastercard)` method because the official card docs expose `payment_method = vm` rather than separate Visa and Mastercard request values.
- The hosted card query-response signature formula is implemented using the documented sorted-parameter signing pattern across the returned signed fields. If Latipay applies a narrower response-signature subset in production, that behavior will need to be aligned against sandbox evidence or provider confirmation.
- Refund support is implemented against the documented refund endpoint, but the documentation does not describe a signed refund-status query API. Ambiguous refund results therefore remain in manual review.
- The provider `order_id` is used operationally, but the plugin does not trust unsigned or mismatched values to override validated order state.
